# 角色按剧本区分 - 深度分析与方案

## 当前系统分析

### 1. 角色数据流

```
剧本解析 (ScriptManager.tsx)
    ↓
解析出角色名称列表
    ↓
用户点击"创建角色" → 创建 CharacterAsset
    ↓
保存到 storage (projectId/assetId)
    ↓
CharacterSidebar 加载所有角色 (按 projectId 筛选)
```

### 2. 关键数据结构

**Asset 基础接口 (types.ts)**

```typescript
export interface Asset {
  id: string;
  projectId: string; // 关联项目
  type: AssetType; // CHARACTER
  name: string;
  prompt: string;
  metadata?: Record<string, any>;
  createdAt: number;
  updatedAt: number;
}
```

**当前问题：Asset 没有 scriptId 字段**

- 角色只关联到项目，不关联到具体剧本
- 多个剧本的角色混在一起

### 3. 角色创建流程分析

**创建位置：** `ScriptManager.tsx` → 角色标签页 → 创建角色按钮

**当前逻辑：**

1. 解析剧本获得角色名称列表
2. 对每个角色名称，用户可以"创建角色"
3. 创建时只传入 `projectId` 和角色名称
4. 没有记录该角色来自哪个剧本

### 4. 角色显示位置

**CharacterSidebar.tsx**

- 显示在分镜管理页面左侧
- 加载项目中所有角色资产
- 按创建时间排序
- 没有剧本分组或筛选功能

---

## 用户需求深度理解

### 用户场景

1. 项目 "test 001" 中有多个剧本（如"剧本A"、"剧本B"）
2. "剧本A" 有角色：主角、反派
3. "剧本B" 有角色： narrato、配角
4. 当前显示：主角、反派、narrato、配角（混在一起）
5. 期望：可以筛选只看"剧本A"的角色

### 核心需求

- **按剧本筛选角色**：只显示特定剧本的角色
- **全部显示选项**：保留当前全部显示的能力
- **直观易用**：在角色列表界面直接操作

---

## 方案对比

### 方案A：给 Asset 添加 scriptId 字段（推荐）

**实施步骤：**

1. 修改 `Asset` 接口，添加 `scriptId?: string`
2. 创建角色时，传入当前剧本的 ID
3. 在 `CharacterSidebar` 添加剧本筛选器
4. 按 `scriptId` 筛选角色

**优点：**

- 数据关系清晰，角色明确属于某个剧本
- 筛选逻辑简单高效
- 符合用户直觉
- 扩展性好（以后可以支持角色跨剧本复用）

**缺点：**

- 需要修改数据结构
- 需要处理现有数据（现有角色没有 scriptId）

**向后兼容策略：**

- `scriptId` 设为可选字段
- 现有角色没有 `scriptId` 时，显示在"未分类"或"全部"中
- 用户可以选择"全部显示"查看所有角色

---

### 方案B：通过剧本内容反向关联

**实施步骤：**

1. 解析剧本时，记录"剧本包含哪些角色名称"
2. 显示角色时，检查该角色名称是否在当前剧本中
3. 按匹配度筛选

**优点：**

- 不需要修改数据结构
- 实现较快

**缺点：**

- 逻辑复杂，性能差（需要遍历所有剧本）
- 同一角色名在不同剧本中可能是不同角色
- 角色改名后关联失效
- 不符合用户直觉

---

### 方案C：角色创建时复制到剧本

**实施步骤：**

1. 创建角色时，在剧本的 `metadata` 中记录角色ID列表
2. 显示时读取剧本的 `metadata.characterIds`
3. 只显示这些角色

**优点：**

- 不需要修改 Asset 结构
- 角色可以属于多个剧本

**缺点：**

- 数据冗余（角色信息分散）
- 删除角色时需要同步更新所有剧本
- 逻辑复杂

---

## 推荐方案：方案A

**核心理由：**

1. **数据关系清晰** - 角色明确属于某个剧本
2. **实现简单** - 只需添加一个字段和筛选器
3. **性能好** - 直接按字段筛选，无需复杂计算
4. **用户体验好** - 直观易懂
5. **扩展性好** - 未来可以支持角色跨剧本复用（一个角色关联多个剧本）

---

## 详细实施计划

### 阶段1：数据结构修改

**修改文件：** `types.ts`

```typescript
export interface Asset {
  id: string;
  projectId: string;
  type: AssetType;
  name: string;
  prompt: string;
  scriptId?: string; // 新增：关联剧本ID
  metadata?: Record<string, any>;
  createdAt: number;
  updatedAt: number;
}
```

### 阶段2：角色创建逻辑修改

**修改文件：** `ScriptManager.tsx` 和角色创建相关组件

- 创建角色时，传入当前剧本的 `scriptId`
- 保存到 asset 数据

### 阶段3：角色列表界面优化

**修改文件：** `CharacterSidebar.tsx`

**新增功能：**

1. **剧本筛选器**
   - 下拉框："全部剧本" / "剧本A" / "剧本B" ...
   - 默认显示"全部剧本"
2. **筛选逻辑**
   - 选择"全部剧本"：显示所有角色（包括没有 scriptId 的）
   - 选择特定剧本：只显示 `scriptId` 匹配的角色

3. **视觉优化**
   - 在角色卡片上显示所属剧本名称（可选）
   - 没有 scriptId 的角色显示"未分类"标签

### 阶段4：现有数据处理

**处理策略：**

- 现有角色没有 `scriptId`，在"全部剧本"模式下显示
- 用户可以编辑现有角色，手动选择所属剧本（可选功能）

## 风险评估

### 风险1：现有角色数据

**风险：** 现有角色没有 `scriptId`，筛选后可能"消失"
**缓解：**

- "全部剧本"模式显示所有角色
- 添加"未分类"选项显示无 scriptId 的角色

### 风险2：角色跨剧本复用

**风险：** 一个角色可能在多个剧本中出现
**缓解：**

- 当前方案中，每个剧本创建独立的角色资产
- 未来可以扩展为 `scriptIds: string[]` 支持多剧本关联

### 风险3：剧本删除后角色孤儿

**风险：** 删除剧本后，角色的 `scriptId` 指向不存在的数据
**缓解：**

- 删除剧本时，可以选择"同时删除角色"或"保留角色"
- 保留的角色显示在"未分类"中

---

## 结论

**推荐实施方案A**，因为它：

1. 数据关系清晰直观
2. 实现简单可靠
3. 用户体验好
4. 向后兼容性好
5. 扩展性强

这个方案能从根本上解决"角色按剧本区分"的需求，同时不影响现有功能。
