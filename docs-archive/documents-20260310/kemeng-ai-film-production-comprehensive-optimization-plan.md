## 一、项目现状诊断

### 1.1 核心问题总结

经过多轮深度分析，项目存在以下结构性问题：

#### 架构层面

- **功能冗余**: 分步解析与完整解析逻辑重复，维护成本高
- **代码分散**: 同一功能（批处理）在多个方法中重复实现
- **遗留代码**: ChromaDB/VectorMemory移除不彻底，仍有残留引用
- **配置混乱**: 并发参数、超时参数分散在多处，缺乏统一管理

#### 性能层面

- **调用次数过多**: 339字符文本触发20+次LLM调用
- **串行瓶颈**: 完整解析使用逐个提取，而非批处理
- **模型响应**: 单次调用42秒，远超行业平均水平
- **无快速路径**: 短文本和长文本使用相同流程

#### 产品层面

- **用户体验**: "一键解析"vs"分步解析"让用户困惑
- **功能堆砌**: 智能记忆、迭代优化等功能未验证实际价值
- **缺乏监控**: 无法量化解析质量和性能瓶颈

### 1.2 已完成的优化（保留）

| 优化项           | 效果                              | 状态    |
| ---------------- | --------------------------------- | ------- |
| 全局上下文合并   | 110s → 42s (61%提升)              | ✅ 保留 |
| 并发配置优化     | concurrency:1→3, delay:1000→100ms | ✅ 保留 |
| 性能监控日志     | 详细的阶段耗时统计                | ✅ 保留 |
| Provider参数修复 | 正确的provider传递                | ✅ 保留 |
| ChromaDB清理     | 移除VectorMemory核心代码          | ✅ 保留 |

### 1.3 需要删除的功能

| 功能                  | 删除理由                         | 影响             |
| --------------------- | -------------------------------- | ---------------- |
| 分步解析              | 与完整解析逻辑重复，增加维护成本 | 用户需适应新流程 |
| VectorMemoryToggle UI | ChromaDB已移除，功能失效         | 界面简化         |
| 智能记忆开关          | 功能已移除，UI残留               | 界面简化         |
| 迭代优化引擎          | 未验证实际效果，增加复杂度       | 代码简化         |
| 逐个提取方法          | 批处理已存在，代码重复           | 维护简化         |

## 三、核心优化策略

### 3.1 架构重构: 单一解析引擎

**删除分步解析，统一使用完整解析流程**

```typescript
// 删除前: 两个入口
parseScript()  // 完整解析
parseStage()   // 分步解析

// 删除后: 单一入口
parseScript(content: string, config: ParseConfig): Promise<ParseState>
```

**统一批处理策略**

```typescript
// 删除前: 两套实现
extractCharacter(); // 逐个提取
extractAllCharactersWithContext(); // 批处理

// 删除后: 单一实现
extractAllCharactersWithContext(); // 只保留批处理
```

### 3.2 智能路径选择

**根据文本长度自动选择解析策略**

```typescript
interface ParseStrategy {
  type: 'fast' | 'standard' | 'chunked';
  maxTokens: number;
  batchSize: number;
  useGlobalContext: boolean;
}

function selectStrategy(wordCount: number): ParseStrategy {
  if (wordCount < 800) {
    return { type: 'fast', maxTokens: 4000, batchSize: 10, useGlobalContext: false };
  } else if (wordCount < 5000) {
    return { type: 'standard', maxTokens: 8000, batchSize: 5, useGlobalContext: true };
  } else {
    return { type: 'chunked', maxTokens: 12000, batchSize: 3, useGlobalContext: true };
  }
}
```

### 3.3 批处理优化

**角色批处理**

- 每批最多5个角色
- 超过5个分多批并发
- 批处理失败自动降级到逐个提取

**场景批处理**

- 每批最多5个场景
- 与角色批处理并行执行

**分镜生成**

- 使用ConcurrencyLimiter控制并发
- 每场景2-4个分镜，批量生成

### 3.4 短文本快速路径

**<800字文本优化**

```typescript
// 单次调用完成全部解析
const result = await llmProvider.generateStructured({
  prompt: shortTextPrompt,
  schema: {
    metadata: ScriptMetadataSchema,
    characters: ScriptCharacterArraySchema,
    scenes: ScriptSceneArraySchema,
    shots: ShotArraySchema,
  },
});
```

**预期效果**: 1-2次调用完成，60-90秒

## 五、技术规范

### 5.1 代码规范

**单一职责原则**

- 每个方法只做一件事
- 批处理和逐个提取不共存
- 策略选择与实际执行分离

**可测试性**

- 所有策略可独立测试
- 降级机制可模拟验证
- 性能指标可量化测量

**可配置性**

- 所有阈值外部可配置
- 策略参数可动态调整
- 支持A/B测试

### 5.2 错误处理

**降级策略**

```typescript
try {
  return await batchExtract();
} catch (e) {
  console.warn('Batch failed, falling back to individual');
  return await individualExtract();
}
```

**重试机制**

- 网络错误：指数退避重试3次
- 模型错误：立即降级
- 超时错误：记录并跳过

### 5.3 性能监控

**关键指标**

- 总解析时间
- 每个阶段耗时
- LLM调用次数
- Token使用量
- 缓存命中率

**日志规范**

```typescript
console.log(`[ScriptParser] Stage: ${stage}, Duration: ${duration}ms, Tokens: ${tokens}`);
```
