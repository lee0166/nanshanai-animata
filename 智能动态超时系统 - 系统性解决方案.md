# 智能动态超时系统 - 系统性解决方案

**文档版本**: v1.0  
**创建时间**: 2026-04-12  
**问题来源**: localhost-200.log 场景提取超时（180 秒）  
**适用范围**: 小说解析全流程（1 千字 -50 万字）

---

## 一、问题本质与生产场景分析

### 1.1 当前问题表象

**日志数据**（localhost-200.log）：
```
角色提取：prompt 2983 chars, 耗时 60 秒，✅ 成功
场景提取：prompt 3345 chars, 超时 180 秒，❌ 失败
```

**表面原因**：
- 默认超时 90 秒太短
- 场景提取需要更多时间

**深层问题**：
- 无历史数据时，超时计算完全靠猜
- 忽略内容长度、任务复杂度、模型特性
- 无法适应 1 千字到 50 万字的真实生产场景

---

### 1.2 真实生产环境分析

#### 场景 1：超短篇小说（<3000 字）
- **典型场景**：微小说、段子、短故事
- **解析特点**：角色少（1-3 个）、场景少（1-5 个）
- **超时需求**：60-120 秒
- **当前问题**：90 秒默认值勉强够用，但场景提取仍有风险

#### 场景 2：中篇小说（3000-30000 字）
- **典型场景**：知乎盐选、番茄小说单章、公众号连载
- **解析特点**：角色中等（5-15 个）、场景中等（5-20 个）
- **超时需求**：120-360 秒
- **当前问题**：**这是当前系统的重灾区**，90 秒默认值完全不够

#### 场景 3：长篇小说（30000-100000 字）
- **典型场景**：单章大章、中篇完整故事
- **解析特点**：角色多（15-50 个）、场景多（20-50 个）
- **超时需求**：360-600 秒
- **当前问题**：必然超时，系统无法处理

#### 场景 4：超长篇小说（>100000 字）
- **典型场景**：完整长篇小说、多章节合集
- **解析特点**：角色极多（50+ 个）、场景极多（50+ 个）
- **超时需求**：600-1200 秒
- **当前问题**：系统完全无法处理，需要分章节解析

## 二、系统性解决方案设计

### 2.1 核心设计原则

#### 原则 1：零配置启动（Zero-Config Bootstrapping）
- **问题**：新环境无历史数据时，超时计算靠猜
- **方案**：基于内容长度和 prompt 长度的预估模型
- **目标**：第一次调用就能给出合理的超时时间

#### 原则 2：自适应学习（Adaptive Learning）
- **问题**：固定配置无法适应不同模型、不同内容
- **方案**：基于历史响应的移动平均 + 方差分析
- **目标**：越用越准确，10 次调用后预测准确度>85%

#### 原则 3：多维度因子（Multi-Dimensional Factors）
- **问题**：单一因子无法准确预测
- **方案**：内容长度 × 任务复杂度 × 模型特性 × 历史性能
- **目标**：综合准确率>80%

#### 原则 4：分级保护（Tiered Protection）
- **问题**：单一超时限制无法应对极端情况
- **方案**：最小超时 → 预测超时 → 最大超时的三层保护
- **目标**：既不浪费等待时间，也不 premature timeout

#### 原则 5：生产级容错（Production-Grade Fault Tolerance）
- **问题**：超时后直接失败，用户无法继续
- **方案**：自动重试 + 熔断降级 + 优雅失败
- **目标**：单次失败不影响整体流程

### 2.2 数学模型设计

#### 2.2.1 超时计算公式

```typescript
timeout = clamp(
  baseTime × contentFactor × complexityFactor × modelFactor × safetyFactor,
  minTimeout,
  maxTimeout
)

// 其中：
// - baseTime: 基础时间（基于历史或预估）
// - contentFactor: 内容长度因子 [0.5 - 3.0]
// - complexityFactor: 任务复杂度因子 [0.8 - 5.0]
// - modelFactor: 模型特性因子 [0.8 - 1.5]
// - safetyFactor: 动态安全系数 [1.3 - 3.0]
// - minTimeout: 任务类型最小超时
// - maxTimeout: 任务类型最大超时
```

#### 2.2.2 内容长度因子

**设计思路**：
- 2000 字以内：基准值 1.0
- 2000-10000 字：线性增长
- 10000-50000 字：对数增长（增长放缓）
- 500000 字以上：饱和增长（避免过大）

**数学公式**：
```typescript
contentFactor = f(contentLength) =
  1.0,                                    if contentLength <= 2000
  1.0 + (L-2000)/8000,                    if 2000 < L <= 10000
  2.0 + log10(L/10000) × 0.8,            if 10000 < L <= 50000
  2.8 + log10(L/50000) × 0.4,            if L > 50000

// 限制范围：[0.5, 3.0]
```

**实际值示例**：
| 内容长度 | 因子值 | 说明 |
|---------|-------|------|
| 1000 字 | 1.0 | 短篇小说 |
| 5000 字 | 1.38 | 中篇单章 |
| 10000 字 | 2.0 | 中篇完整 |
| 50000 字 | 2.8 | 长篇单章 |
| 100000 字 | 3.0 | 长篇完整（饱和） |

#### 2.2.3 任务复杂度因子

**设计思路**：
- 基于任务类型的基础复杂度
- 基于输出项数量的对数增长
- 基于 prompt 长度的线性修正

**数学公式**：
```typescript
complexityFactor = baseComplexity × (1 + log10(itemCount)) × promptFactor

// 其中：
// - baseComplexity: 任务基础复杂度
// - itemCount: 输出项数量（角色数/场景数/分镜数）
// - promptFactor: 1 + max(0, (promptLength - 3000) / 10000)
```

**基础复杂度表**：
```typescript
const BASE_COMPLEXITY = {
  METADATA: 0.8,      // 元数据提取
  CHARACTER: 1.0,     // 角色提取
  SCENE: 1.5,         // 场景提取（含 visualPrompt）
  ITEM: 0.9,          // 物品提取
  SHOTS: 2.5,         // 分镜生成
  GENERAL: 1.0,       // 通用
};
```

**实际值示例**（场景提取，7 个场景，prompt 3345 字）：
```typescript
complexityFactor = 1.5 × (1 + log10(7)) × (1 + 345/10000)
                 = 1.5 × 1.85 × 1.03
                 = 2.86
```

#### 2.2.4 模型特性因子

**设计思路**：
- 基于历史响应时间的移动平均
- 基于响应时间方差的稳定性评分
- 基于超时率的惩罚因子

**数学公式**：
```typescript
// 1. 移动平均响应时间（EMA）
avgTime = 0.7 × avgTime_prev + 0.3 × currentTime

// 2. 稳定性评分（基于方差）
variance = |currentTime - avgTime| / avgTime
stabilityScore = max(0, 1 - variance)

// 3. 模型因子
modelFactor = 1.0 + (1 - stabilityScore) × 0.5

// 稳定性差时，modelFactor 接近 1.5
// 稳定性好时，modelFactor 接近 1.0
```

#### 2.2.5 动态安全系数

**设计思路**：
- 响应时间越长，安全系数越低（避免过度保守）
- 稳定性越差，安全系数越高（保护性超时）
- 任务越重要，安全系数越高（分镜>场景>角色）

**数学公式**：
```typescript
baseFactor = 
  2.0,    if avgTime <= 60s
  1.8,    if 60s < avgTime <= 90s
  1.5,    if 90s < avgTime <= 120s
  1.3,    if avgTime > 120s

// 稳定性修正
if stabilityScore < 0.7:
  baseFactor += 0.3

// 任务类型修正
if taskType == SHOTS:
  baseFactor += 0.2

// 最终限制
safetyFactor = clamp(baseFactor, 1.3, 3.0)
```

### 2.3 架构设计

#### 2.3.1 组件结构

```
DynamicTimeoutCalculator 2.0
├── ContentLengthAnalyzer（内容长度分析器）
│   ├── calculateContentFactor()
│   └── estimateBaseTime()
│
├── TaskComplexityAnalyzer（任务复杂度分析器）
│   ├── calculateComplexityFactor()
│   └── getBaseComplexity()
│
├── ModelPerformanceProfiler（模型性能画像器）
│   ├── updateProfile()
│   ├── getStabilityScore()
│   └── getModelFactor()
│
├── AdaptiveTimeoutEngine（自适应超时引擎）
│   ├── calculateSafetyFactor()
│   ├── calculateTimeout()
│   └── clampTimeout()
│
└── TimeoutMonitor（超时监控器）
    ├── recordSuccess()
    ├── recordFailure()
    └── getStats()
```

#### 2.3.2 数据流

```
1. 调用前：getTimeout(contentLength, itemCount, promptLength)
   ↓
2. 内容分析：contentFactor = analyzeContent(contentLength)
   ↓
3. 复杂度分析：complexityFactor = analyzeComplexity(taskType, itemCount, promptLength)
   ↓
4. 模型画像：modelFactor = getModelFactor(modelId)
   ↓
5. 安全系数：safetyFactor = calculateSafetyFactor(avgTime, stability)
   ↓
6. 最终计算：timeout = baseTime × contentFactor × complexityFactor × modelFactor × safetyFactor
   ↓
7. 限制范围：clampedTimeout = clamp(timeout, minTimeout, maxTimeout)
   ↓
8. 返回结果
```

## 四、风险评估与应对

### 风险 1：过度保守导致等待时间过长

**风险描述**：
- 新系统可能预测过大的超时时间
- 用户需要等待更长时间

**应对措施**：
- 设置合理的 maxTimeout 上限
- 添加进度提示（"已耗时 X 秒，预计还需 Y 秒"）
- 支持用户手动取消

**监控指标**：
- 平均等待时间 < 180 秒
- 用户取消率 < 5%

---

### 风险 2：预测不准确导致仍然超时

**风险描述**：
- 复杂场景下预测模型可能失效
- 仍然会出现超时

**应对措施**：
- 保留自动重试机制
- 超时后自动增加 50% 重试
- 记录失败案例用于优化

**监控指标**：
- 超时率 < 10%
- 重试成功率 > 60%

---

### 风险 3：性能开销过大

**风险描述**：
- 复杂计算可能增加延迟
- 影响用户体验

**应对措施**：
- 优化计算逻辑（O(1) 复杂度）
- 缓存模型性能数据
- 异步更新画像

**监控指标**：
- 超时计算耗时 < 10ms
- 不影响整体解析流程

---

### 风险 4：配置参数调优困难

**风险描述**：
- 大量参数需要调优
- 不同环境表现不同

**应对措施**：
- 提供默认优化配置
- 支持运行时动态调整
- 基于监控数据自动调优

**监控指标**：
- 参数调整频率 < 1 次/周
- 调优后效果提升明显
