## 优化策略

### 策略一：保守优化（推荐）

**工作量**: ⭐⭐ 中等  
**风险**: ⭐ 极低  
**预计时间**: 2-3 小时  
**回滚难度**: 容易

**核心思路**: 只修改与主题色相关的颜色，保留中性色和语义化颜色

#### 阶段1: 替换 Indigo → Primary（高优先级）

将硬编码的 indigo 颜色替换为 primary，确保主题一致性

**涉及文件**:

- `views/Settings.tsx` - 约 15 处 indigo
- `contexts/AppContext.tsx` - 约 5 处 indigo
- `components/Layout.tsx` - 约 2 处 indigo
- `components/AssetList.tsx` - 约 2 处 indigo
- `components/ProjectDetail/Shared/ImageGenerationPanel.tsx` - 约 2 处 indigo
- `components/ProjectDetail/GenerationForm.tsx` - 约 1 处 indigo
- `components/ResourcePicker.tsx` - 约 1 处 indigo

**修改示例**:

```tsx
// 修改前
className="text-indigo-600 dark:text-indigo-400"
bg-indigo-50/30 dark:bg-indigo-900/10
shadow-indigo-500/30

// 修改后
className="text-primary"
bg-primary/10
shadow-primary/30
```

#### 阶段2: 替换 Blue → Primary（中优先级）

将功能性蓝色替换为主题色

**涉及文件**:

- `views/ShotManager.tsx` - 约 8 处 blue
- `contexts/ToastContext.tsx` - 1 处 blue (info toast)

**修改示例**:

```tsx
// 修改前
text-blue-500 hover:text-blue-600

// 修改后
text-primary hover:text-primary-700
```

#### 阶段3: 优化阴影效果（低优先级）

统一阴影颜色为主题色

**涉及文件**:

- `views/Settings.tsx`
- `contexts/AppContext.tsx`
- `components/Layout.tsx`
- `components/AssetList.tsx`
- `components/ProjectDetail/Shared/ImageGenerationPanel.tsx`
- `components/ProjectDetail/GenerationForm.tsx`

### 策略二：全面优化

**工作量**: ⭐⭐⭐⭐ 高  
**风险**: ⭐⭐⭐ 中  
**预计时间**: 1-2 天  
**回滚难度**: 困难

**核心思路**: 全面重构色彩系统，包括中性色、语义化颜色

#### 需要修改的内容：

1. **中性色系统** - 将所有 slate/gray 调整为暖色调中性色
2. **语义化颜色** - 重新定义 success/error/warning/info 的颜色
3. **渐变效果** - 添加橙红色渐变背景
4. **图标系统** - 统一图标颜色规范

**风险点**:

- 大面积修改容易引入视觉回归
- 语义化颜色修改可能影响用户认知（如红色=错误）
- 测试覆盖困难

## 推荐方案：分阶段保守优化

### 为什么推荐保守方案？

1. **风险可控** - 只修改主题相关颜色，不影响中性色和语义化颜色
2. **工作量合理** - 约 2-3 小时即可完成
3. **易于回滚** - 每个阶段独立，出问题可单独回滚
4. **视觉改善明显** - 能解决"突兀"问题，提升品牌一致性
5. **不影响功能** - 纯样式修改，零功能风险

### 具体实施计划

#### 阶段1: Indigo → Primary（30分钟）

```bash
# 修改范围
- text-indigo-* → text-primary
- bg-indigo-* → bg-primary/10 或 bg-primary/20
- border-indigo-* → border-primary
- shadow-indigo-* → shadow-primary
```

#### 阶段2: Blue → Primary（20分钟）

```bash
# 修改范围
- text-blue-500/600 → text-primary
- hover:text-blue-* → hover:text-primary-700
```

#### 阶段3: 阴影统一（15分钟）

```bash
# 将所有 shadow-indigo-500/30 改为 shadow-primary/30
```

#### 阶段4: 验证测试（30分钟）

- 检查所有页面按钮颜色
- 检查设置页面配置区域
- 检查加载状态颜色
- 检查阴影效果

## 风险评估与缓解

### 潜在风险

| 风险           | 可能性 | 影响 | 缓解措施                     |
| -------------- | ------ | ---- | ---------------------------- |
| 误改其他代码   | 低     | 中   | 使用精确搜索替换，逐文件检查 |
| 颜色对比度不足 | 中     | 中   | 修改后验证可访问性           |
| 视觉回归       | 低     | 低   | 分阶段修改，每阶段验证       |
| 遗漏修改       | 中     | 低   | 使用 grep 复查               |

### 质量保证措施

1. **Git 备份** - 修改前创建分支
2. **逐文件修改** - 不批量替换，逐个文件确认
3. **实时预览** - 修改后立即在浏览器验证
4. **对比检查** - 修改前后截图对比

## 结论

### 可行性: ✅ 完全可行

**您的顾虑是合理的，但可以通过以下方式解决**:

1. **采用保守方案** - 只修改主题色相关，不动中性色和语义色
2. **分阶段实施** - 每个阶段独立，降低风险
3. **使用精确替换** - 避免误删代码
4. **实时验证** - 每改一个文件立即验证

```
1. 创建 git 分支备份
2. 执行阶段1 (Indigo → Primary)

4. 执行阶段2 (Blue → Primary)
5. 浏览器验证
6. 执行阶段3 (阴影统一)
7. 全面验证
8. 合并分支
```
