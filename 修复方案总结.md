# 小说解析分镜生成问题修复方案

## 📋 修复概述

基于日志分析（localhost-2.log），发现小说解析过程中分镜生成失败的核心问题是**配置参数严重偏离行业标准**，导致：
- 1257 字生成 150 个分镜（应为 30-40 个）
- 每批 15 个分镜导致超时失败（应为 5 个）
- 连续 2 次失败后系统放弃，生成 0 个分镜

## ✅ 已修复内容

### 1. 分镜密度配置（scriptParser.ts）

#### 修复内容：
```typescript
// 修复前
shotDensityShort: 10,  // ❌ 每 200 字 10 个分镜

// 修复后
shotDensityShort: 4,   // ✅ 每 200 字 4 个分镜
```

#### 修复依据：
- **抖音短剧标准**：1-2 分钟视频 20-40 个分镜 → 10-20 个/分钟
- **快手短剧标准**：2-3 分钟视频 30-50 个分镜 → 10-17 个/分钟
- **影视行业经验**：平均镜头长度 2-4 秒 → 15-30 个/分钟
- **本项目语速**：200 字/分钟 → 4 个/200 字 = 12 个/分钟（符合标准）

#### 预期效果：
```
1257 字 ÷ 200 字/分钟 = 6.3 分钟
6.3 分钟 × 4 个/分钟 = 25 个分镜（基础）
考虑平台特性 × 1.3 = 33 个分镜

对比：现在 150 个 → 修复后 33 个，减少 78% ✅
```

### 3. 批量大小配置（scriptParser.ts）

#### 修复内容：
```typescript
// 修复前

  initial: 15,  // ❌ 每批 15 个分镜
  min: 8,
  max: 25,
};

// 修复后
const BATCH_SIZE_CONFIG = {
  initial: 5,   // ✅ 每批 5 个分镜
  min: 3,
  max: 10,
};
```

#### 修复依据：
**竞品对比**：
| 工具 | 批量大小 | 设计理念 |
|------|---------|---------|
| Runway ML | 5-10 个 | 少即是多 |
| Pika Labs | 4-8 个 | 小批量多次 |
| Stable Video | 1-4 个 | 极致质量 |
| Midjourney | 4 个 | 4 宫格输出 |
| **本项目（修复前）** | 15 个 | ❌ 贪多嚼不烂 |
| **本项目（修复后）** | 5 个 | ✅ 保证质量 |

**技术依据**：
- **LLM 注意力机制**：超过 10 个对象，注意力分散
- **超时风险**：
  - 15 个/批：需要 193 秒（超时失败）
  - 5 个/批：需要 60 秒（安全）
- **失败成本**：
  - 15 个/批：失败 2 次损失 30 个分镜，触发中止
  - 5 个/批：失败 2 次损失 10 个分镜，可继续
