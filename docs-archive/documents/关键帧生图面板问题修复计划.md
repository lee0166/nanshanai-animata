# 关键帧生图面板问题修复计划（根本性解决方案）

## 背景与问题

1. 原有模型配置没有区分文生图和参考图生图
2. 需要从根本上解决，适配后期新增的不同模型
3. 当前代码层面的临时兼容不是长久之计

## 根本性解决方案

### 方案：模型配置增加能力标识

**核心思路：** 修改模型配置数据结构，为每个模型明确标识其能力。

#### 方案A：修改 DEFAULT_MODELS 结构（推荐）

在 `config/models.ts` 中的 `DEFAULT_MODELS` 为每个模型添加明确的能力标识：

```typescript
export const DEFAULT_MODELS: ModelConfig[] = [
  {
    id: 'volc-img-seedream-4.0',
    name: 'Seedream 4.0',
    provider: 'volcengine',
    type: 'image',
    modelId: 'doubao-seedream-4-0-250828',
    capabilities: {
      supportsReferenceImage: true, // 明确标识支持参考图
      maxReferenceImages: 10,
    },
  },
  {
    id: 'volc-img-seedream-3.0-t2i',
    name: 'Seedream 3.0 T2I',
    provider: 'volcengine',
    type: 'image',
    modelId: 'doubao-seedream-3-0-t2i-250715',
    capabilities: {
      supportsReferenceImage: false, // 明确标识不支持参考图
    },
  },
  // ... 其他模型
];
```

**优势：**

- 清晰明确，每个模型的能力一目了然
- 新增模型时只需在 DEFAULT_MODELS 中添加配置
- 代码层面只需读取配置，无需复杂匹配逻辑

**劣势：**

- 需要修改配置文件
- 用户已有的自定义模型需要重新配置

#### 方案B：运行时模型配置增加能力设置

在模型设置界面，让用户为每个生图模型手动选择是否支持参考图：

```typescript
// 在设置中增加模型能力配置
interface ModelCapabilityConfig {
  modelId: string;
  supportsReferenceImage: boolean;
  maxReferenceImages?: number;
}

// 用户可以在设置界面配置每个模型的能力
```

**优势：**

- 用户可以自由配置自定义模型
- 不需要修改代码中的 DEFAULT_MODELS
- 灵活适配任何新模型

**劣势：**

- 需要开发设置界面
- 用户需要手动配置

#### 方案C：混合方案（推荐）

结合方案A和方案B：

1. **DEFAULT_MODELS 提供默认能力配置** - 内置已知模型的能力
2. **用户可以在设置中覆盖** - 对于自定义模型或需要调整的情况
3. **代码优先使用用户配置，其次使用默认配置**

```typescript
// 获取模型能力的优先级：
// 1. 用户自定义配置（如果有）
// 2. DEFAULT_MODELS 中的配置（如果匹配）
// 3. 默认值（支持参考图）

const getModelCapabilities = (model: ModelConfig) => {
  // 1. 检查用户自定义配置
  const userConfig = settings.modelCapabilities?.find(m => m.modelId === model.modelId);
  if (userConfig) return userConfig;

  // 2. 检查 DEFAULT_MODELS
  const defaultModel = DEFAULT_MODELS.find(m => m.modelId === model.modelId || m.id === model.id);
  if (defaultModel?.capabilities) return defaultModel.capabilities;

  // 3. 默认值
  return { supportsReferenceImage: true, maxReferenceImages: 5 };
};
```

## 建议实施方案

**推荐方案C（混合方案）**，因为它：

1. **从根本上解决** - 明确标识模型能力
2. **向后兼容** - 已有模型可以继续使用
3. **灵活扩展** - 支持用户自定义模型
4. **渐进式升级** - 不需要一次性修改所有配置
