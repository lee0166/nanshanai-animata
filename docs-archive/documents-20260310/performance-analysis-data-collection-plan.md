# 剧本解析性能问题数据分析计划

> **计划原则**: 先收集数据，再分析问题，最后才考虑优化方案  
> **当前阶段**: 数据收集与分析  
> **禁止事项**: 在数据分析完成前，不修改任何项目代码

---

## 一、问题背景

### 当前观察到的现象

- 339字短文本解析到40%耗时约6分钟
- 全局上下文提取耗时110秒（近2分钟）
- 用户反馈：同类型应用中不会有这么长的用时

### 需要回答的核心问题

1. **时间花在哪里了？** 模型响应 vs 代码逻辑 vs 网络延迟
2. **是项目代码问题还是模型本身慢？** 需要量化数据
3. **短文本和长文本的差异？** 当前架构是否合理
4. **与行业对比如何？** 豆包 Seed 1.6 的正常响应时间应该是多少

## 二、数据收集阶段

### Phase 1: 添加详细耗时监控（3天）

**目标**: 在关键路径添加精确的时间戳记录，定位时间消耗点

#### Task 1.1: LLMProvider 层耗时监控

**只读分析**: 阅读 `services/ai/providers/LLMProvider.ts`
**需要记录的数据**:

- 请求发送时间戳
- 响应接收时间戳
- 实际网络耗时（端到端）
- JSON解析耗时
- 错误重试次数和耗时

**输出**: 在计划文档中记录需要监控的具体代码位置

#### Task 1.2: ScriptParser 层调用链监控

**只读分析**: 阅读 `services/scriptParser.ts`
**需要记录的数据**:

- 每个 Stage 的开始/结束时间
- 每个 LLM 调用的开始/结束时间
- 并发队列等待时间
- 批量提取 vs 逐个提取的实际调用次数

**输出**: 绘制调用链时间线图

#### Task 1.3: 网络层监控

**只读分析**: 阅读 `services/ai/providers/BaseProvider.ts`
**需要记录的数据**:

- DNS解析时间
- 连接建立时间
- 首字节时间（TTFB）
- 内容下载时间
- 代理转发额外耗时（开发环境）

**输出**: 网络耗时分解报告

### Phase 2: 外部基准测试（2天）

**目标**: 排除项目代码影响，测试模型本身的响应速度

#### Task 2.1: 直接 API 测试

**执行方式**: 使用 curl 直接调用火山引擎 API
**测试内容**:

```bash
# 测试1: 简单请求（339字文本，简单prompt）
curl -X POST https://ark.cn-beijing.volces.com/api/v3/chat/completions \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "doubao-seed-1-6-251015",
    "messages": [{"role": "user", "content": "分析以下文本的类型和基调..."}],
    "max_tokens": 1000
  }' \
  -w "\nDNS: %{time_namelookup}\nConnect: %{time_connect}\nTTFB: %{time_starttransfer}\nTotal: %{time_total}\n"
```

**需要测试的场景**:

- 简单请求（100 tokens 输入，100 tokens 输出）
- 中等请求（1000 tokens 输入，500 tokens 输出）
- 复杂请求（8000 tokens 输入，2000 tokens 输出）
- 对比测试：豆包-lite-32k vs 豆包-seed-1.6 vs DeepSeek-V3

**输出**: 各模型响应时间基准数据

#### Task 2.2: 开发环境 vs 生产环境对比

**测试内容**:

- 开发环境（通过 Vite 代理）的响应时间
- 生产环境构建后的响应时间（如果可部署）
- 代理转发带来的额外延迟

**输出**: 环境差异分析报告

### Phase 3: 同类型应用对比（2天）

#### Task 3.2: 豆包官方文档调研

## 三、数据分析阶段

### Phase 4: 数据整理与可视化（2天）

#### Task 4.1: 时间消耗分解

**分析方法**:

```
总耗时 = 网络延迟 + 模型推理时间 + 代码处理时间 + 队列等待时间

网络延迟 = DNS解析 + TCP连接 + TLS握手 + 请求发送 + 响应接收
模型推理时间 = 首字节时间(TTFB) - 网络延迟
代码处理时间 = JSON解析 + 数据验证 + 缓存操作
队列等待时间 = 并发限制导致的等待
```

**输出**: 时间消耗饼图和时序图

#### Task 4.2: 瓶颈识别

**分析方法**:

- 找出耗时占比最大的环节
- 识别异常耗时（如：某个环节耗时远超预期）
- 对比短文本和长文本的差异

**输出**: 瓶颈优先级列表

#### Task 4.3: 根因分析

**分析方法**:

- 5 Whys 分析法
- 对比测试数据分析
- 行业基准对比

**输出**: 根因分析报告

## 四、决策阶段

### Phase 5: 优化方案制定（2天）

**前提**: 基于 Phase 1-4 的数据分析结果

#### 可能的优化方向（待数据验证）

**方向A: 如果是模型本身慢**

- 考虑更换更快的模型（豆包-lite-32k）
- 优化 Prompt 减少输出长度
- 使用流式响应提升感知速度

**方向B: 如果是网络延迟高**

- 生产环境直接连接 API（绕过代理）
- 使用 HTTP/2 连接复用
- 考虑后端托管方案

**方向C: 如果是代码逻辑问题**

- 提高并发限制（concurrency: 1 → 3）
- 移除调用延迟（callDelay: 1000 → 0）
- 优化批量提取逻辑

**方向D: 如果是架构设计问题**

- 短文本快速路径
- 异步预加载
- 缓存策略优化

#### Task 5.1: 方案评估

**评估维度**:

- 预期性能提升（基于数据估算）
- 实现成本
- 风险
- 收益/成本比

**输出**: 优化方案对比表

#### Task 5.2: 决策文档

**内容**:

- 基于数据的结论
- 推荐的优化方案
- 实施计划
- 预期效果

**输出**: 决策文档
