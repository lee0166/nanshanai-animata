### 3.2 节点2：文本预处理

**源文件**：`services/textCleaner.ts`

#### 3.2.1 TextCleaner.clean() 完整处理流程

```typescript
static clean(text: string, language?: LanguageType): string {
  // 1. 统一换行符 (\r\n, \r → \n)
  cleaned = cleaned.replace(/\r\n/g, '\n').replace(/\r/g, '\n');

  // 2. 去除控制字符（保留换行、制表符）
  cleaned = cleaned.replace(/[\x00-\x08\x0B\x0C\x0E-\x1F\x7F-\x9F]/g, '');

  // 3. 合并多余空行（3个及以上 → 2个）
  cleaned = cleaned.replace(/\n{3,}/g, '\n\n');

  // 4. 去除行首行尾空白
  cleaned = cleaned.replace(/^[ \t]+|[ \t]+$/gm, '');

  // 5. 统一空格
  cleaned = cleaned.replace(/[ \t]+/g, ' ');

  // 6. 全角标点转半角（仅对中文/混合语言）
  // 包含：，。！？；：（）[]""''等

  // 7. 省略号统一为...
  cleaned = cleaned.replace(/[…．．．。。。]/g, '...');

  // 8. 破折号统一为--
  cleaned = cleaned.replace(/[—−－]/g, '--');

  // 9. 智能引号处理
  cleaned = cleaned.replace(/[""]/g, '"').replace(/['']/g, "'");

  // 10. 语言特定处理
  // 中文：确保标点后有空格，中英文间有空格
  // 英文：确保标点后有空格
  // 混合：综合处理
}
```

// IterativeRefinementEngine.ts:161-210
async refine(
  metadata: ScriptMetadata,
  characters: ScriptCharacter[],
  scenes: ScriptScene[]
): Promise<IterativeRefinementResult> {
  // 迭代优化循环
  for (let iteration = 1; iteration <= this.config.maxIterations; iteration++) {
    // Step 1: 一致性检查
    const consistencyResult = this.consistencyChecker.check({...});

    // Step 2: 质量评估
    const qualityResult = this.qualityEvaluator.evaluate({...});

    // Step 3: 生成修正动作
    const refinementResult = this.refinementEngine.refine({...});

    // Step 4: 应用修正
    currentMetadata = this.applyRefinements(currentMetadata, refinementResult);

    // 检查是否满足停止条件
    if (qualityScore >= this.config.targetQualityScore) break;
    if (improvement < this.config.minImprovementThreshold) break;
  }
}

// storage.ts:112-145
private async lock<T>(key: string, fn: () => Promise<T>): Promise<T> {
  let resolveUnlock: () => void;
  const newLock = new Promise<void>(resolve => { resolveUnlock = resolve; });

  const currentLock = this.locks.get(key) || Promise.resolve();
  this.locks.set(key, currentLock.then(() => newLock));

  try {
    await currentLock.catch(() => {});
    return await fn();
  } finally {
    resolveUnlock!();
  }
}

## 五、潜在技术瓶颈分析

### 5.1 API依赖风险

| 瓶颈             | 严重程度 | 缓解措施                      |
| ---------------- | -------- | ----------------------------- |
| LLM API调用超时  | 高       | CircuitBreaker + 超时重试     |
| API速率限制(429) | 高       | callDelay=1000ms + 并发控制=1 |
| API Key无效      | 中       | 配置校验 + 错误提示           |

### 5.2 Token消耗风险

| 阶段        | Token消耗   | 优化措施             |
| ----------- | ----------- | -------------------- |
| Ultra-Short | ~500-1000   | 单次调用，极简Prompt |
| Fast        | ~2000-5000  | 并行提取，减少60-70% |
| Standard    | ~5000-20000 | 动态批量 + Token优化 |
| Chunked     | ~20000+     | 语义分块 + 缓存      |

### 5.3 内存与性能风险

| 风险点     | 描述            | 当前处理                |
| ---------- | --------------- | ----------------------- |
| 大文件解析 | PDF/DOCX > 10MB | 50MB限制 + 流式读取     |
| 长文本处理 | > 10000字       | Chunked Path语义分块    |
| 状态持久化 | 中断恢复        | saveState/loadState机制 |
| 内存泄漏   | useEffect闭包   | isMountedRef检查        |

### 5.4 数据一致性风险

| 风险点       | 描述             | 处理机制            |
| ------------ | ---------------- | ------------------- |
| 部分解析失败 | 中断后状态不一致 | 幂等设计 + 增量更新 |
| 并发修改     | 多标签页同时编辑 | 互斥锁机制          |
| 缓存过期     | 旧数据覆盖新数据 | TTL + 版本校验      |
