## 根本原因分析

### 1. 模型响应时间过长

- 单次 API 调用平均耗时: **~25秒**
- 最慢的调用: **82秒** (Shot 4)
- 这是 **doubao-seed-1-6-flash-250828** 模型的固有问题

### 2. API 调用次数过多

- Metadata: 1次
- Global Context: 4次
- Characters: 3次
- Scenes: 12次
- Shots: 15次
- **总计: 35次 API 调用**

### 3. 串行调用

- 所有调用都是串行执行，没有并行化
- 35次调用 × 25秒 = 875秒（约14.6分钟）

---

## 结论

### 这是模型问题，不是项目设计问题

1. **模型响应慢**: doubao-seed-1-6-flash-250828 模型平均响应时间 25秒，远超正常水平（正常应 < 5秒）

2. **调用次数合理**: 35次调用对于完整解析流程是正常的
   - 每个角色、场景、分镜都需要单独提取
   - 这是为了保证提取质量

3. **项目设计优化空间**:
   - 可以并行化角色/场景提取（但受限于模型并发限制）
   - 可以缓存全局上下文结果
   - 可以使用更快的模型

---

## 优化建议

### 短期方案（立即见效）

1. **更换更快的模型**
   - 使用 doubao-lite-4k 或其他轻量级模型
   - 预计可将单次调用时间从 25秒降至 5秒
   - 总时间从 20分钟降至 4分钟

2. **禁用全局上下文**（临时方案）
   - 可节省 4次 API 调用（约100秒）
   - 在 `ScriptParser` 配置中设置 `useGlobalContext: false`

### 中期方案（需要开发）

1. **并行化提取**
   - 并行提取多个角色/场景
   - 预计可节省 30-50% 时间

2. **增加缓存机制**
   - 缓存全局上下文结果
   - 避免重复提取相同内容

3. **优化分镜生成**
   - 批量生成多个分镜
   - 减少 API 调用次数

### 长期方案（架构优化）

1. **流式解析**
   - 边解析边展示结果
   - 提升用户体验

2. **异步解析**
   - 后台异步完成解析
   - 用户无需等待
