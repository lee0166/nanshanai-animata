# 控制台日志代码问题分析

## 发现的问题

### 问题1：分镜生成 max_tokens 不足（严重）

**证据**：

```
LLMProvider.ts:85 [LLMProvider] Max Tokens: 4000
LLMProvider.ts:120 [LLMProvider] Completion tokens: 9306
```

**问题**：

- 分镜生成请求的 `max_tokens` 设置为 4000
- 但实际生成了 9306 tokens
- 响应被截断，导致 JSON 不完整
- LLM 可能需要重试或生成不完整的内容

**影响**：

- 分镜生成时间极长（82秒）
- 响应被截断，可能导致解析失败
- 需要多次重试，增加总耗时

**修复建议**：
增加分镜生成的 `max_tokens` 到 10000 或更高。

---

### 问题2：分镜生成数量过多（中等）

**证据**：

```
scriptParser.ts:1374 [ScriptParser] Generated 15 shots
scriptParser.ts:1374 [ScriptParser] Generated 11 shots
scriptParser.ts:1374 [ScriptParser] Generated 15 shots
...
```

**问题**：

- 每个场景生成了 10-15 个分镜
- 12个场景共生成 151 个分镜
- 但只有 15 次分镜 API 调用

**分析**：
看起来分镜是批量生成的（每次调用生成多个分镜），这本来是优化。
但每个分镜的详细描述导致 token 数量爆炸。

**影响**：

- 单次调用生成过多内容
- token 数量超过限制
- 响应时间极长

**修复建议**：

1. 限制每个场景的分镜数量（如最多 5-8 个）
2. 减少每个分镜的描述长度
3. 分批生成，避免单次调用生成过多分镜

---

### 问题3：Prompt 中包含完整剧本内容（中等）

**证据**：

```
LLMProvider.ts:88 [LLMProvider] Request body: {
  ...
  "content": "\n请为以下场景生成分镜脚本。\n\n【场景信息】\n场景名称: 忠仆救主\n场景描述: ...\n\n【剧本原文】\n大靖王朝，御史大夫沈砚因弹劾丞相谋逆，被诬陷下狱，沈家满门抄斩，独留幼子沈清辞被忠仆所救，隐于民间。十年后，沈清辞化名苏砚，以布衣身份入京城，蛰伏于吏部当差，伺机翻案。\n\n他聪慧过人，行事低调，很快凭借精准的断案能力得到吏部尚书赏识，却也..."
}
```

**问题**：

- 每个分镜生成请求都包含完整的剧本原文
- 剧本原文 339 字符，但每次请求都重复发送
- 15次分镜请求 = 15 × 339 = 5085 字符的重复内容

**影响**：

- 增加不必要的 token 消耗
- 增加 API 成本
- 可能让 LLM 困惑（过多无关信息）

**修复建议**：
只发送与当前场景相关的剧本片段，而不是完整剧本。

---

### 问题4：API 调用没有超时控制（轻微）

**证据**：

```
LLMProvider.ts:103 [LLMProvider] Request completed in 82286ms
```

**问题**：

- 单次调用耗时 82 秒，没有超时中断
- 如果模型卡住，可能会无限等待

**修复建议**：
添加超时控制，如 30 秒超时后重试或报错。

---

## 总结

### 确认的项目代码问题

1. **分镜生成 max_tokens 不足**（严重）
   - 设置 4000，实际需要 9000+
   - 导致响应截断和重试

2. **分镜生成数量过多**（中等）
   - 每个场景 10-15 个分镜
   - 单次调用生成过多内容

3. **Prompt 包含完整剧本**（中等）
   - 重复发送完整剧本
   - 浪费 token 和带宽

4. **缺少超时控制**（轻微）
   - 可能导致无限等待

### 与模型问题的关系

- 模型响应慢是主要问题（25秒/次）
- 但项目代码问题加剧了这个问题：
  - max_tokens 不足导致重试
  - 分镜过多导致单次调用时间过长
  - 重复内容增加 token 消耗

### 修复优先级

1. **高优先级**：增加分镜生成的 max_tokens
2. **中优先级**：限制分镜数量，优化 prompt
3. **低优先级**：添加超时控制
