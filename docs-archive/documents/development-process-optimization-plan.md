# 开发流程优化与测试修复规划

## 一、建立全局回滚点规则（SKILL）

### 1.1 背景与必要性

在复杂项目修复和优化任务中，经常需要：

- 修改多个相互关联的文件
- 进行多轮迭代调试
- 面临不确定的副作用风险

**没有回滚点的风险**：

- 修改导致系统无法运行
- 无法快速恢复到稳定状态
- 调试过程混乱，无法追踪变更

### 1.2 建议的SKILL规则

**SKILL名称**: `checkpoint-before-execution`

**触发条件**:

- 涉及3个及以上文件的修改
- 修改核心服务层代码（services/目录）
- 修改类型定义（types.ts）
- 涉及正则表达式、解析逻辑等复杂算法
- 用户明确标记为"复杂任务"

**执行流程**:

```
用户请求复杂任务
    ↓
AI识别需要设置回滚点
    ↓
提示用户："此任务涉及多处修改，建议先设置回滚点。是否继续？"
    ↓
用户确认后
    ↓
执行 git stash / git commit 创建回滚点
    ↓
开始实施修改
    ↓
每完成一个阶段，询问用户是否继续
```

### 1.3 SKILL实现方案

**文件位置**: `.trae/skills/checkpoint-before-execution/SKILL.md`

**内容结构**:

1. 识别复杂任务的判断标准
2. 与用户交互的确认流程
3. 创建回滚点的具体操作
4. 不同场景下的回滚策略

## 二、现有测试失败问题评估与修复规划

### 2.1 问题现状

当前测试失败统计：
| 测试文件 | 失败数 | 原因 | 影响程度 |
|---------|-------|------|---------|
| scriptParser.test.ts | 3 | semanticChunker mock 问题 | 中 |
| scriptParser.integration.test.ts | 1 | semanticChunker mock 问题 | 中 |

**失败详情**:

1. `semanticChunker.chunkSync is not a function` - mock 未正确设置
2. 错误消息不匹配 - 期望消息与实际消息不一致

### 2.2 影响分析

#### 对生产环境的影响

- **低** - 实际代码有防御性检查，不会崩溃
- **中** - 如果启用语义分块功能，可能无法正常工作
- **高** - 长期不修复会导致技术债务累积

#### 对开发流程的影响

- 新团队成员看到测试失败会感到困惑
- 无法通过测试快速验证代码变更
- CI/CD 流程可能受阻

### 2.3 修复方案

#### 方案A：修复测试 Mock（推荐）

**步骤1**: 修复 `semanticChunker` 的 mock

```typescript
// 在测试文件中
const mockSemanticChunker = {
  chunkSync: vi.fn().mockReturnValue([{ content: 'chunk1' }, { content: 'chunk2' }]),
};
```

**步骤2**: 修复错误消息断言

```typescript
// 更新期望的错误消息
expect(() => parser.extractJSON(response)).toThrow('Failed to parse JSON');
```

**工作量**: 2-3小时
**风险**: 低
**收益**: 高

#### 方案B：重构测试架构

**目标**: 建立更健壮的测试隔离机制

**改进点**:

1. 统一 mock 工厂函数
2. 建立测试数据生成器
3. 添加测试环境检查
4. 完善测试文档

**工作量**: 1-2天
**风险**: 中
**收益**: 长期收益高

### 2.4 建议的修复优先级

```
P0 (立即修复):
  - semanticChunker mock 问题
  - 错误消息断言更新

P1 (本周内):
  - 添加测试环境文档
  - 配置沙箱白名单
```

## 三、长期开发规范建议

### 3.1 测试规范

**必须遵循的原则**:

1. **隔离性** - 所有外部依赖必须 mock
2. **确定性** - 测试不依赖环境、时间、随机数
3. **独立性** - 测试之间不相互依赖
4. **可读性** - 测试代码应该清晰表达意图

**禁止行为**:

- 直接调用外部 API
- 依赖文件系统（除非使用内存文件系统）
- 依赖环境变量（除非明确 mock）
- 测试之间相互依赖

### 3.2 代码审查清单

**修改前检查**:

- [ ] 是否涉及多个文件？
- [ ] 是否修改核心逻辑？
- [ ] 是否设置回滚点？

**修改后检查**:

- [ ] 单元测试是否通过？
- [ ] 是否添加/更新测试用例？
- [ ] 文档是否更新？

### 3.3 文档规范

**必须包含的内容**:

1. 测试环境配置说明
2. 常见错误及解决方案
3. 沙箱环境限制说明
4. 如何运行测试
