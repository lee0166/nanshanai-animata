## 🔴 P0-1：场景内容提取不完整

### **问题描述**

**现象**：
```
[ScriptParser] Extracted 0 chars for scene: 林薇的办公室
```

**影响**：
- 该场景无法生成分镜或分镜质量下降
- 影响分镜总数（目标 80 个，实际 26 个）
- 用户体验差，需要手动补充场景内容

### **根因分析**

**原因**：
1. 小说中`林薇的办公室`场景没有详细描述
2. 场景定位逻辑依赖关键词匹配，未处理"场景存在但无内容"的情况
3. 原代码直接返回空字符串或默认 1000 字符，缺乏智能处理

### **修复方案**

**修复文件**：`services/scriptParser.ts`

**新增方法**：`generateDefaultSceneDescription()`


## 🔴 P0-2：分镜生成提前终止

### **问题描述**

**现象**：
```
[ScriptParser] Final result: 26 shots (25 key + 1 optional) in 8 batches
目标：80 shots (56 key + 24 optional)
完成率：32.5%（26/80）
```

**影响**：
- 分镜数量不足，影响后续视频生成
- 用户需要手动补充分镜，增加工作量
- 用户体验差，认为系统功能不完整

### **根因分析**

**原因**：
1. **maxBatches 配置过低**：默认值为 8，生成 8 批后强制终止
2. **缺乏完成率检查**：即使只完成了 32.5%，也会正常终止
3. **缺少诊断日志**：无法判断终止原因

**计算验证**：
```
目标：80 个分镜
批量：5 个/批（第一批）+ 3 个/批（后续）
8 批最多生成：5 + 3×7 = 26 个分镜
完成率：26/80 = 32.5%
```

### **修复方案**

**修复文件**：`services/scriptParser.ts`

**修改 1：增加 maxBatches**

```typescript
// 修复前
const maxBatches = this.parserConfig.maxShotBatches || 8;

// 修复后
const maxBatches = this.parserConfig.maxShotBatches || 20;  // 从 8 增加到 20，避免提前终止
const targetCompletionRate = 0.9;  // 目标完成率 90%
```

**修改 2：添加完成率检查**

```typescript
while (
  allShots.length < targetShots &&
  currentBatch < maxBatches &&
  consecutiveFailures < maxConsecutiveFailures
) {
  currentBatch++;
  const remainingShots = targetShots - allShots.length;
  
  // ✅ 新增：检查是否已达到目标完成率
  const currentCompletionRate = allShots.length / targetShots;
  if (currentCompletionRate >= targetCompletionRate && remainingShots <= 5) {
    console.log(
      `[ScriptParser] Reached target completion rate (${(currentCompletionRate * 100).toFixed(1)}%), stopping generation`
    );
    break;
  }
  
  // ... 继续生成逻辑
}
```

**修改 3：增加诊断日志**

```typescript
// 记录分镜生成完成统计
const completionRate = allShots.length / targetShots;
console.log(
  `[ScriptParser] Shot generation completed: ${allShots.length}/${targetShots} (${(completionRate * 100).toFixed(1)}%), batches: ${currentBatch}`
);

if (completionRate < 0.8) {
  console.warn(
    `[ScriptParser] Warning: Completion rate is low (${(completionRate * 100).toFixed(1)}%), ` +
      `consider increasing maxBatches or checking scene content`
  );
}
```

### **修复效果**

**预期效果**：
- ✅ 分镜生成完成率从 32.5% → 90%+
- ✅ maxBatches 从 8 增加到 20，提供足够的生成空间
- ✅ 达到 90% 完成率且剩余≤5 个分镜时智能终止
- ✅ 低完成率（<80%）时输出警告，便于诊断问题

**示例输出**：

## 🟡 P1-1：进度计算错误

### **问题描述**

**现象**：
```
[ProgressTracker] Calling onProgressCallback with: stage=shots, progress=57%
期望：100%（分镜阶段完成）
实际：57%
```

**影响**：
- 用户无法准确了解解析进度
- 可能误以为解析未完成
- 进度条显示不准确，影响用户体验

### **根因分析**

**原因**：
1. **阶段权重配置不合理**：shots 阶段权重仅 20%，但实际是最后的主要阶段
2. **其他阶段权重未归零**：refinement、budget、episode_planning 等阶段在短文本解析中实际不执行，但仍占用权重

**权重分析**：
```
修复前权重配置：
- metadata: 12%
- characters: 20%
- scenes: 20%
- items: 5%
- shots: 20%  ← 问题：占比过低
- refinement: 3%  ← 问题：实际不执行
- budget: 2%  ← 问题：实际不执行
- episode_planning_*: 7%  ← 问题：实际不执行

总权重：100%
但实际执行的阶段权重：12+20+20+5+20 = 77%
导致进度显示不准确
```

### **修复方案**

**修复文件**：`services/parsing/ProgressTrackerConfig.ts`

**修改权重配置**：

```typescript
export const DEFAULT_STAGE_WEIGHTS: Record<ParseStage, number> = {
  idle: 0,
  metadata: 0.12,     // 12% - 元数据提取
  characters: 0.2,    // 20% - 角色分析
  scenes: 0.2,        // 20% - 场景分析
  items: 0.05,        // 5% - 物品提取
  shots: 0.43,        // 43% - 分镜生成（占比最高，因为是最主要的阶段）
  refinement: 0.0,    // 0% - 跳过结果优化（合并到分镜生成）
  budget: 0.0,        // 0% - 跳过时长预算（合并到分镜生成）
  episode_planning_phase1: 0.0,  // 0% - 跳过分集规划 Phase 1
  episode_planning: 0.0,         // 0% - 跳过分集规划
  episode_planning_phase2: 0.0,  // 0% - 跳过分集规划 Phase 2
  coherence_check: 0.0,          // 0% - 跳过连贯性检查
  completed: 0,
  error: 0,
};
```

**权重验证**：
```
12% + 20% + 20% + 5% + 43% = 100% ✅
```

### **修复效果**

**预期效果**：
- ✅ 进度计算准确率从 57% → 100%
- ✅ 分镜阶段完成时，总进度显示 100%

## 🟡 P1-2：响应时间波动大

### **问题描述**

**现象**：
```
Batch 1: 125.7 秒
Batch 2: 74.9 秒
Batch 3: 90.0 秒
Batch 4: 90.4 秒
Batch 5: 81.4 秒
Batch 6: 92.8 秒
Batch 7: 73.9 秒
Batch 8: 70.1 秒

波动范围：70-126 秒（1.8 倍）
```

**影响**：
- 动态超时计算需要更多样本才能稳定
- 第一批响应时间过长，触发批量下调
- 用户体验不一致

### **优化方案**

**说明**：响应时间波动问题已在之前的优化中通过**动态批量调整机制**解决。

**已有优化**：
```typescript
// 动态批量调整（已在 scriptParser.ts 中实现）
private adjustBatchSize(
  currentSize: number,
  responseTime: number,
  config: {
    min: number;
    max: number;
    adjustThreshold: {
      decrease: number;
      increase: number;
    };
  }
): number {
  if (responseTime > config.adjustThreshold.decrease) {
    // 响应慢：减小批次
    const newSize = Math.max(config.min, currentSize - 3);
    return newSize;
  }

  if (responseTime < config.adjustThreshold.increase) {
    // 响应快：增大批次
    const newSize = Math.min(config.max, currentSize + 2);
    return newSize;
  }

  return currentSize;
}
```

**优化效果**：
- ✅ 第一批响应慢时自动从 5 个降至 3 个
- ✅ 后续批次根据响应时间动态调整
- ✅ 避免超时失败，提高成功率

### **进一步优化建议**

**如需进一步优化响应时间稳定性，可考虑**：
