# 后端日志清理与优化计划

## 当前问题分析

### 1. 日志文件过大

- 当前日志文件大小：365KB
- 行数：6593行
- 主要问题：大量重复的"加载失败"错误日志

### 2. 错误类型统计

- 模型未激活错误（404 ModelNotOpen）：多次重复
- 网络请求失败：大量重复（行370-395）
- 这些重复错误导致日志文件膨胀

## 优化方案

### 阶段1：清理现有日志文件

1. **备份当前日志**
   - 将现有日志备份为 `后端日志.txt.bak`
   - 便于后续问题排查

2. **清空当前日志**
   - 清空 `后端日志.txt` 内容
   - 重新开始记录

### 阶段2：优化日志记录机制

1. **减少重复错误日志**
   - 对相同的错误进行去重
   - 只记录首次错误和错误计数

2. **调整日志级别**
   - 将调试日志改为 `debug` 级别
   - 生产环境只记录 `warn` 和 `error`

3. **添加日志轮转**
   - 限制单个日志文件大小（如10MB）
   - 自动归档旧日志

### 阶段3：修复根本问题

1. **模型未激活问题**
   - 在UI中提示用户激活模型
   - 添加模型状态检查

2. **网络请求失败**
   - 添加重试机制
   - 优化错误提示

## 执行步骤

### 步骤1：备份并清空日志

```bash
# 备份
Copy-Item "后端日志.txt" "后端日志.txt.bak"

# 清空
Clear-Content "后端日志.txt"
```

### 步骤2：优化日志代码

- 修改 `BaseProvider.ts` 减少重复错误日志
- 修改 `LLMProvider.ts` 优化错误处理
- 添加错误去重机制

### 步骤3：添加用户提示

- 当模型未激活时，在UI中显示友好提示
- 引导用户去火山方舟控制台激活模型

## 预期效果

- 日志文件大小控制在合理范围（<1MB）
- 错误信息清晰，便于排查
- 重复错误自动去重
- 用户能及时了解模型状态

## 注意事项

1. 清理日志前务必备份
2. 保留关键错误信息
3. 确保问题可追溯
